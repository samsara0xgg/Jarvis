## 待解决问题

### TTS 声音质量（当前最大痛点）
四个 TTS 引擎都试过了，没一个满意：
- **OpenAI TTS**: 中文发音像外国人，逐句有停顿
- **Azure**: 声音机械，播音腔
- **MiniMax**: 质量最好但余额不足
- **Edge TTS**: 免费但没情感

**下一步方案**：
1. 充值 MiniMax（最快解决）
2. 解决逐句TTS停顿问题（合并短句再调API）
3. 探索 CosyVoice 本地部署（Mac GPU 可跑）

### DeepSeek 拒绝情感表达
DeepSeek 的安全训练太强，personality prompt 压不住。
已切换到 GPT-4o-mini 解决。

### LLM 选择结论
| 场景 | 推荐 | 原因 |
|------|------|------|
| 闲聊/情感 | GPT-4o-mini | 遵从 prompt，不拒绝情感 |
| 性价比 | DeepSeek V4 | $0.30/M，但拒绝情感 |
| 中文最强 | DeepSeek V4 | 中文理解最好 |
| 意图路由 | Groq (llama-3.3-70b) | 免费+极快 |

## Session 2 — 记忆系统（同日后续）

### 项目全面审查
- 完整梳理了数据流（麦克风 → ASR → 路由 → LLM → TTS）
- 发现核心问题：小贾是"云端大脑+本地四肢"，记忆存了但从未注入 prompt
- 确定方向：本地大脑 = 记忆系统 + 知识检索 + 减少云端依赖

### 记忆系统设计 + 研究
- 对比 xiaozhi-esp32-server、Mem0、MemGPT/Letta、EMem (ICLR 2025)
- 研究 bge-small-zh-v1.5 vs potion-base-8M vs multilingual-e5-small
- 确定技术栈：FastEmbed ONNX + numpy cosine + SQLite
- 设计记忆生命周期 6 环节：提取 → 去重 → 存储 → 检索 → 注入 → 维护

### 记忆系统实现（2791 行新代码）
- `memory/embedder.py` — bge-small-zh-v1.5 via FastEmbed，线程安全
- `memory/store.py` — SQLite 三表（memories/profiles/episodes），schema 迁移
- `memory/retriever.py` — 多信号检索 + category-dependent 衰减 + access 强化
- `memory/manager.py` — query/save/maintain 完整管线
- 集成到 personality.py / llm.py / jarvis.py / memory_skill.py
- 63 新测试，614 总测试全过

### 关键技术决策
- 6 种 category：identity/preference/relationship/event/task/knowledge
- 去重：key-first 确定性 + embedding 模糊（同 category 0.55 / 跨 category 0.7）+ LLM 终裁
- 检索：<100 条全量注入，≥100 条三层检索
- 衰减：identity 365天 / preference 180天 / event 30天 / task 14天半衰期
- 周维护：每周日 3am 合并重复
