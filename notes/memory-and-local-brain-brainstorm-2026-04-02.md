wod# 记忆层 + 本地大脑 Brainstorm — 2026-04-02

## 核心问题

能不能把大脑拿回本地，同时不影响质量？

**结论：不需要完全替换云端 LLM，而是减少依赖频率。** 聪明地分层，能让 60-70% 的交互不走云端 LLM，同时质量不降。即使还要调云端，给 LLM 更好的上下文（记忆+知识），回答质量也比现在高。

---

## 分层智能架构

```
用户说话
    │
    ▼
Level 0: 规则匹配（瞬时，无模型）彻底
    "几点了" → 直接回答
    "晚安" → 触发自动化规则
    "开客厅灯" → 正则/关键词 → 设备控制
    │ 没匹配到
    ▼
Level 1: 记忆检索（<100ms，本地 embedding）
    "我喜欢喝什么" → 检索用户记忆 → "你说过你喜欢拿铁"
    "我昨天让你干嘛了" → 检索对话摘要 → 回答
    "Python list 排序" → 检索知识库 → 直接给答案
    │ 没找到相关记忆/知识
    ▼
Level 2: 本地 LLM 尝试（可选，Mac 上 Ollama）
    简单问题本地回答，质量够用就不上云
    │ 问题太复杂 / 没有本地 LLM
    ▼
Level 3: 云端 LLM（GPT-4o-mini）
    真正需要推理的问题
    但这次带着完整记忆和知识上下文
    → 质量比现在更高，因为 LLM 知道你是谁、你喜欢什么
```

## 场景可行性分析

| 场景 | 能本地化吗 | 说明 |
|------|-----------|------|
| "几点了" | ✅ 已经本地 | — |
| "开灯" | ✅ 已经本地 | — |
| "我上次说我喜欢什么" | ✅ 可以本地 | 记忆检索，不需要 LLM |
| "明天温哥华天气" | ⚠️ 半本地 | 数据要联网拉，但格式化可以本地做 |
| "帮我总结一下这篇文章" | ❌ 必须云端 | 需要真正的语言理解 |
| "你觉得我该不该换工作" | ❌ 必须云端 | 需要推理+人格 |
| "Python 的 list 怎么排序" | ✅ 可以本地 | 如果有本地知识库 |
| "给我讲个笑话" | ⚠️ 看质量要求 | 本地 LLM 可以但质量差些 |

## 记忆层设计 — 四种记忆

```
┌─────────────────────────────────────────────────────┐
│                    小月的记忆                         │
│                                                      │
│  ① 工作记忆（当前对话）                                │
│     = conversation_store，最近 20 轮                  │
│     现在有 ✅                                         │
│     作用：知道"刚才"聊了什么                            │
│                                                      │
│  ② 用户档案（关于你这个人）                             │
│     = 你是谁、你的习惯、你的偏好                        │
│     现在有一半 ⚠️（存了但不注入 prompt）                │
│     作用：知道"你是谁"                                 │
│                                                      │
│  ③ 经历记忆（过去发生了什么）                            │
│     = 历史对话的摘要、重要事件                           │
│     现在没有 ❌                                        │
│     作用：知道"以前"发生了什么                           │
│                                                      │
│  ④ 世界知识（事实和技能）                               │
│     = 你教她的东西、常识、文档                           │
│     现在没有 ❌                                        │
│     作用：知道"世界"是什么样的                           │
└─────────────────────────────────────────────────────┘
```

### ① 工作记忆 — 已有，但有问题

现状：`ConversationStore` 保存最近 20 轮原始消息，重启后还在。

问题：
- 20 轮之后就完全丢失，没有任何提取
- 一段很重要的对话（"我下周要出差"）过了 20 轮就消失了
- 没有 summary 机制

改进：对话结束时，**自动提取重要信息存入用户档案**。20 轮的原始消息可以丢，但里面的关键信息不能丢。

### ② 用户档案 — 最容易修、收益最大

现状：`UserPreferenceStore` 存了 key-value，`MemorySkill` 可以 remember/recall/forget。但 `_personalize_system()` **没把偏好传给 prompt**。

这是整个项目最大的 bug — 不是代码 bug，是架构 bug。修复只需要两步：
1. `_personalize_system()` 里加载用户偏好，传给 `build_personality_prompt(preferences=...)`
2. 对话结束后自动提取事实存入偏好

修完之后，你说"记住我喜欢喝拿铁"，下一次对话小月就知道你喜欢拿铁。**这个改动 10 行代码，但体验提升最大。**

### ③ 经历记忆 — 中等难度，很有价值

现在完全没有。需要新建。

设计思路：
```
每次对话结束后:
  对话原文 → (用 LLM 或规则) 提取摘要
  → 存成一条 episode:
    {
      "date": "2026-04-02",
      "summary": "Allen 问了股票行情，NVDA 涨了3%。他说下周要出差深圳。",
      "facts_extracted": ["user_travel: 下周出差深圳"],
      "mood": "neutral"
    }

下次对话时:
  system prompt 里注入最近 5 条 episode 摘要
  → 小月知道"昨天你问了股票，上周你出差了"
```

这样即使原始消息被截断，**关键信息以摘要形式永久保留**。

### ④ 世界知识 — 最大的工程，"本地大脑"的核心

让小月从"能干活的工具"变成"有知识的管家"的关键。

两种知识来源：

**a) 主动教的**
- "记住，家里 WiFi 密码是 xxx"
- "记住，Python 的 list.sort() 是原地排序"
- 直接存入知识库

**b) 自己积累的**
- 从对话里提取的事实
- 从新闻/股票数据里沉淀的趋势
- 用户发的文档/链接

**存储和检索方案**：
```
知识条目 → embedding 向量 (本地 all-MiniLM-L6-v2, ~80MB)
                              ↓
用户提问 → embedding → 余弦相似度检索 → top-K 条目
                              ↓
                    注入 LLM prompt 作为上下文
                    或直接模板化回答（不需要 LLM）
```

Mac 上 embedding <50ms。RPi5 上也能跑，稍慢。

## 修完之后的新数据流

```
用户说话
    │
    ▼
识别: ASR + 声纹 (不变)
    │
    ▼
┌─ Level 0: 规则匹配 ─────────────────────────────┐
│  关键词触发 → 直接执行                             │
│  时间查询 → 直接回答                               │
│  设备控制 → 直接执行                               │
└──────────────────────────────────┬───────────────┘
                                   │ 没匹配
                                   ▼
┌─ Level 1: 记忆+知识检索 ────────────────────────┐
│                                                  │
│  1. 检索用户档案 → 有直接相关的？直接回答           │
│     "我喜欢喝什么" → "你说过你喜欢拿铁"            │
│                                                  │
│  2. 检索经历记忆 → 有相关的？回答                   │
│     "我上次让你干嘛" → "昨天你让我查了NVDA的股价"   │
│                                                  │
│  3. 检索知识库 → 有相关知识？回答                   │
│     "WiFi密码多少" → "你告诉过我，是 xxx"           │
│     "list怎么排序" → "用 list.sort() 或 sorted()" │
│                                                  │
└──────────────────────────────────┬───────────────┘
                                   │ 没找到 / 需要推理
                                   ▼
┌─ Level 2/3: LLM (本地或云端) ───────────────────┐
│                                                  │
│  system prompt 里现在有:                          │
│    ✅ 小月人格                                    │
│    ✅ 时段+情绪+情境                               │
│    ✅ 用户档案（"他喜欢拿铁，住温哥华，是程序员"）    │
│    ✅ 最近经历摘要（"昨天聊了股票，上周他出差了"）    │
│    ✅ 检索到的相关知识片段                           │
│    ✅ 对话历史                                     │
│                                                  │
│  → 回答质量比现在高，因为 LLM 知道你是谁             │
│  → 而且调用频率降低了（很多问题 Level 0-1 就解决）   │
└─────────────────────────────────────────────────┘
```

## 实施优先级

| 优先级 | 改动 | 工作量 | 效果 |
|--------|------|--------|------|
| **P0** | 修 `_personalize_system`，注入用户偏好 | 10 行 | 小月立刻"有记忆" |
| **P1** | 对话结束自动提取事实 → 存入用户档案 | ~100 行 | 记忆自动增长 |
| **P2** | 经历记忆：对话摘要 + 注入 prompt | ~200 行 | "昨天你说了什么"能回答 |
| **P3** | 知识库：embedding + 检索 + 注入 | ~500 行 | 本地能回答知识类问题 |
| **P4** | Level 0 扩展：更多规则匹配 | 按需 | 减少云端调用 |

P0 当天可完成。P1-P2 一周内。P3 是更大的工程但核心就是 embedding 检索。
