# 语音架构调研 — 2026-04-02

## 核心发现：为什么 GPT 语音像真人

### 架构差异

```
当前 Jarvis（传统管线）：
麦克风 → ASR(语音转文字) → LLM(思考) → TTS(文字转语音) → 喇叭
              ↑                                    ↑
           文字是瓶颈                          文字是瓶颈

GPT-4o 语音（端到端）：
麦克风 → 音频token → GPT-4o(一个模型搞定一切) → 音频token → 喇叭
                    没有文字中间环节
```

### 文字是信息瓶颈

语音转文字的那一刻，以下信息全部丢失：
- 用户说话的语气（兴奋？疲惫？焦虑？）
- 语调变化（反问？犹豫？强调？）
- 停顿、叹气、笑声
- 说话速度、音量变化

LLM 收到的只是 "今天天气怎么样" 一串字符，不知道用户是开心还是烦躁。
TTS 再从文字生成语音，只能靠语法猜语调 — 语法正确但情感空洞。

GPT-4o 不经过文字。音频 token 保留了语气、情感、呼吸、停顿等所有信息。
模型直接在音频层面思考和生成，回复的情感是**自然涌现的**，不是后期贴上去的。

### 可用方案对比

| 方案 | 质量 | 可行性 | 费用 |
|------|:---:|--------|:---:|
| **OpenAI Realtime API** | 最好 | WebSocket 直连，开发简单 | ~$0.30/分钟（贵） |
| **GLM-4-Voice（智谱）** | 中文很好 | 开源，需要 GPU | 免费 |
| **Moshi（Kyutai）** | 还行 | 开源，可本地跑，~200ms 延迟 | 免费 |
| 当前管线 | 一般 | 已在跑 | 便宜 |

### 对 Jarvis 的意义

当前管线（SenseVoice → Groq → TTS）功能性没问题 — 开灯、查天气、设提醒都能用。
但要达到 GPT 那种"像真人"的效果，不是换个更好的 TTS 就能解决，
而是需要整个架构从"文字管线"变成"端到端音频模型"。

### 决定：明天尝试新架构

计划接入 **OpenAI Realtime API** 作为"高级语音模式"：
- WebSocket 直连，音频进音频出
- 保留现有管线作为"标准模式"（便宜、可离线、有 function calling）
- 两种模式可切换

## 现实路径

```
短期 — 保持现有管线，优化 TTS（CosyVoice 需 GPU，MiniMax 需充值）
中期 — 接入 OpenAI Realtime API 做"高级语音模式"（贵但效果最好）
长期 — 等 GLM-4-Voice / Moshi 成熟后本地部署（免费+隐私）
```
