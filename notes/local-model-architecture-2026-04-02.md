# 本地模型架构构想 — 2026-04-02

## 核心想法

不依赖云端大模型，用本地微调的小模型（Qwen2.5-3B）做主力。
云端 LLM 从"默认"变成"按需" — 只在用户确认后才连接。

## 初版方案（v0，待 brainstorm 验证）

```
用户语音 → SenseVoice ASR
  → 本地 Qwen-3B（微调版）
     ├─ 意图分类 + 参数提取 + 生成回复
     ├─ 调用 skill 执行
     ├─ 不确定 → "这个我不太确定，要帮你查一下吗？"
     └─ 用户确认 → 云端 LLM
  → TTS 播报
```

好处：零月费、零延迟、真隐私、离线可用、人格是训练出来的
风险：3B 天花板、微调需要数据、RPi5 速度慢

---

## Brainstorm v1 结论（3轮后）

三层架构 > 单一本地模型：
```
第一层：规则引擎（零延迟）— "开灯""几点了" 关键词直接执行
第二层：本地 Qwen2.5-1.5B（~1秒）— 意图分类 + JSON
第三层：云端 LLM（按需，用户确认后）— 复杂对话
```

关键判断：
- 1.5B 比 3B 更务实（RPi5 内存安全）
- 先验证裸模型，不急着微调
- 分层比单一模型好
- TTS 是独立问题

---

## Brainstorm v2 结论（8轮后）

### 架构（不变）
```
第一层：规则引擎（零延迟）— 精确匹配，覆盖 ~30-40%
第二层：本地模型（~1-3秒）— 意图分类 + JSON，覆盖 ~50%
第三层：云端 LLM（自动升级，不用每次确认）— 复杂对话 ~10-20%
```

### 修正的判断
- 1.5B **裸模型不够**，需要微调。开发用 3B 验证，部署用微调 1.5B
- 本地模型在 RPi5 上**比 Groq 还慢**（1-3s vs 300ms），优势不在速度
- 核心价值是**"可进化"** — 小月的权重里有你的习惯
- 不要每次确认云端升级，**自动升级，只确认高风险操作**
- 第一层规则引擎覆盖率**别高估**，稍微模糊就得过模型
- **用 GPT-4o 批量生成训练数据**来训练本地模型

### 投入产出评估
- 短期（1周投入）：离线可用 + 无限额 → 对当前场景改善不大
- 长期（持续使用）：可进化 + 真正私有 → 这是云端永远给不了的
- 结论：**长期项目值得投入**

### 执行顺序
```
Day 1: Mac 上 ollama qwen2.5:3b 测裸模型意图分类准确率
Day 2-3: 收集/生成训练数据 (500-1000条)
Day 4: Mac 上 LoRA 微调
Day 5-6: 集成到小月 + RPi5 部署
```
